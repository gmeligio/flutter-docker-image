## Context

`readme.md` is code-generated by `docs/build.mjs` (Node stdlib only) from
`config/version.json`, and the same file is pushed as the Docker Hub description
for `flutter-android` and `flutter-web` (and, once #521 lands, `flutter-windows`).
CI runs `mise run docs` then `git diff --exit-code`, so any hand edit or generator
drift fails the build. The generator today emits: a 7-badge row (3 fixed + 2 per
image for android/web), a generated TOC (`toc()`), a Features block with per-image
tool lists, Running Containers (registry tables + inline snippets for GitHub
Actions/GitLab/Gitea/Forgejo), Tags, Building Locally, Roadmap, and an inline FAQ.
The first `docker run` is at line 58 of 162. Windows is published and public
(289 pulls) but absent from the generated README.

Constraints: no new dependencies (Node stdlib + `cue` only); `config/version.json`
stays the single source of truth; relocated docs must be static markdown (the
minimal-toolchain guardrail). The `version.json` `windows` block already exists
(git, VS Build Tools) but the restructure intentionally drops per-image tool
tables, so it is not rendered.

## Goals / Non-Goals

**Goals:**
- Quick-start above the fold: one-line description + copy-paste usage before any
  reference/contributor content.
- Single Flutter version badge; one pulls badge per platform (android, web,
  windows).
- Windows as a first-class platform (badge, registry table, correct usage).
- Relocate Building Locally → `docs/contributing.md`, FAQ → `docs/faq.md`; remove
  TOC, per-image tool tables, and Roadmap.
- Keep the Flutter version stated in README prose.

**Non-Goals:**
- Changing any published image, tag, or Dockerfile.
- Fixing the windows Docker Hub description sync (issue #521 — separate change;
  this one only makes the synced content describe Windows).
- Adding a Windows `examples/*.yml` file (out of scope; the README snippet is
  inline). The four named CI backends are unchanged.
- Reintroducing any templating engine or doc framework.

## Decisions

**D1 — Single version badge via `static/v1` from `${flutter}`, not `docker/v`.**
All three `docker/v` badges resolve to the identical `v3.44.6`; three is pure
redundancy. A `static/v1` badge reading the manifest `${flutter}` and linking to
the Flutter release archive is truthful (it is the Flutter version, not a
per-image version), pairs with the adjacent `channel` badge (same link target),
and has no Docker Hub push-lag dependence. *Alternative considered:* relabel one
`docker/v/flutter-android` badge as generic `version` — rejected, it arbitrarily
privileges android and misleads if one image lags a release.

**D2 — Generalize `dockerBadge` usage; drop `version` callers.** The existing
helper `dockerBadge(name, kind, label)` already parameterizes the label. Keep it
for the three `pulls` badges (add `flutter-windows`), remove the two `version`
callers, and add the `static/v1` version badge inline. Minimal change to a proven
helper.

**D3 — Delete the `toc()` function and its call.** The spec no longer requires a
TOC and best practice discourages it for a short README. Removing the function
(not just the call) keeps the generator honest — no dead code.

**D4 — Windows usage snippet uses `runs-on: windows-2025`, not `container:`.**
Windows containers cannot run under the Linux `container:` field on ubuntu
runners. Reusing the `ghWorkflow()` helper (which hardcodes `runs-on: ubuntu-22.04`
+ `container:`) for Windows would emit a snippet that cannot work. Windows gets its
own snippet shape. *Alternative:* generalize `ghWorkflow()` to take a runner/mode
— deferred; a second bespoke snippet is simpler than parameterizing for one case.

**D5 — Relocated content is hand-written static markdown, appended/created once.**
Building Locally moves into the existing static `docs/contributing.md`; the FAQ
becomes a new static `docs/faq.md`. Neither is generated, so the manifest version
is not interpolated into them — the Building Locally example keeps a literal
version, matching how `windows.md`/`contributing.md` are already static. The
README links to both. This respects the minimal-toolchain guardrail (no new
generation) at the cost of the Building Locally example version being static (see
R2).

**D6 — Roadmap removed, not relocated.** It is stale (lists Windows as future
though it ships) and duplicates Issues/Milestones. No migration target needed.

## Risks / Trade-offs

- **R1 — Docker Hub description regression for android/web.** The restructured
  README becomes the synced description. → Mitigation: shorter is safe under the
  25 KB Docker Hub limit; verify the rendered description still leads with what/how.
  No behavior change to the sync itself.
- **R2 — Static Building Locally version drifts from the manifest.** Once moved to
  `docs/contributing.md`, the `flutter_version=X.Y.Z` example is no longer
  regenerated. → Mitigation: phrase it as an illustrative example (as the current
  README already does) rather than an authoritative version; the authoritative
  version stays in `config/version.json` and the generated README prose.
- **R3 — Windows snippet correctness.** An inline `windows-2025` example that
  doesn't actually run misleads users. → Mitigation: base it on the published
  image's real usage; it is documentation-only (not CI-executed), so keep it
  minimal and faithful to how the image is pulled.
- **R4 — Removing a whole capability (`image-tool-inventory`) is aggressive.** →
  Mitigation: the REMOVED delta carries an explicit Migration (versions via
  `flutter doctor -v` / `version.json`); the decision-relevant Flutter version is
  retained.

## Automated Test Strategy

The critical path is "generator output matches committed files," already enforced
by the existing docs-in-sync CI check (`mise run docs` then `git diff --exit-code`
in `update-docs.yml`). No new test infrastructure is needed.

- **Regeneration determinism**: `mise run docs` must produce a `readme.md`
  identical to the committed one — the existing `git diff --exit-code` gate covers
  this and will fail the PR on any drift.
- **Content assertions (manual + review-time)**: after regeneration, confirm the
  generated `readme.md` (a) states the manifest Flutter version in prose, (b)
  contains android/web/windows sections and one pulls badge each, (c) contains no
  TOC / tool table / Roadmap, and (d) links to `docs/faq.md` and
  `docs/contributing.md`. These map 1:1 to the spec scenarios.
- **Link check**: the two new relative links (`docs/faq.md`,
  `docs/contributing.md` Building Locally anchor) must resolve — verified by
  inspection; the repo has no automated link checker and this change does not add
  one.
- No unit tests are added for `build.mjs` (it has none today; the diff gate is the
  contract). This is a deliberate scope choice consistent with the existing
  generator.

## Observability

Failures surface at generation and CI time, never silently to an end user:

- **Generator failure** (e.g. a template typo) → `node docs/build.mjs` throws /
  exits non-zero → `mise run docs` fails → PR red. Loud.
- **Drift** (committed README not regenerated) → `git diff --exit-code` fails with
  the existing "run `mise run docs`" instruction. Loud.
- **Silent-failure risk**: a broken *relative link* in the generated or relocated
  markdown would not fail CI. This is the one path that could ship unnoticed →
  mitigated by review-time inspection of the two new links (R3/link check above).
- Docker Hub description sync failures are surfaced by the `update-description`
  job in `release.yml` (out of scope here) — this change does not alter that job's
  error paths.

## Migration Plan

1. Edit `docs/build.mjs`: badges (D1/D2), remove `toc()` + call (D3), rewrite body
   (drop Features tool tables, Roadmap, Building Locally, inline FAQ; add Windows
   section + snippet D4; keep Flutter version in prose).
2. Create `docs/faq.md`; append Building Locally to `docs/contributing.md` (D5).
3. Run `mise run docs`; commit the regenerated `readme.md`.
4. `git diff --exit-code` (locally and in CI) must be clean.

Rollback: revert the commit; `readme.md` and `docs/build.mjs` return together
since the README is generated from the generator in the same commit.
