## MODIFIED Requirements

### Requirement: README presents exact predownloaded versions and ready-to-use examples

`readme.md` SHALL be code-generated by `docs/build.mjs` from `config/version.json`
and SHALL cover **every published image** — `flutter-android`, `flutter-web`, and
`flutter-windows`. The README SHALL be structured as a quick-start: a one-line
description and a copy-paste usage snippet SHALL appear above the fold, before any
reference or contributor material. The README SHALL state the manifest Flutter
version in prose (not only in a badge). For each image the README SHALL render a
Running Containers registry table and a usage snippet whose image tag equals the
manifest Flutter version; the `flutter-windows` snippet SHALL use
`runs-on: windows-2025` (Windows containers cannot run under the Linux
`container:` pattern). The README SHALL name all four supported CI platforms
(GitHub Actions, GitLab CI, Gitea, Forgejo) and link to the `examples/`
directory. The README SHALL NOT include a generated table of contents, per-image
main-tool tables, a Roadmap section, inline Building Locally instructions, or an
inline FAQ; Building Locally SHALL live in `docs/contributing.md` and the FAQ
SHALL live in `docs/faq.md`, linked from the README.

The experience context is a CI engineer reading only the README who identifies an
image and copies a working usage snippet in seconds, and a Windows user who finds
`flutter-windows` presented as a first-class platform — without the README
drifting from the manifest.

#### Scenario: All three images are presented from the manifest

- **WHEN** `readme.md` is generated
- **THEN** it contains a `flutter-android`, a `flutter-web`, and a
  `flutter-windows` section
- **AND** the manifest Flutter version appears in the README prose

#### Scenario: Usage snippets use the current tag

- **WHEN** `readme.md` is generated for manifest Flutter version `X.Y.Z`
- **THEN** the flutter-android, flutter-web, and flutter-windows usage snippets
  reference their images at tag `X.Y.Z`
- **AND** the flutter-windows snippet uses `runs-on: windows-2025`

#### Scenario: Quick-start is above the fold

- **WHEN** a reader views `readme.md`
- **THEN** a one-line description and a copy-paste usage snippet appear before any
  Building Locally, FAQ, or contributor content

#### Scenario: Relocated and removed sections

- **WHEN** `readme.md` is generated
- **THEN** it contains no table of contents, no per-image main-tool table, and no
  Roadmap section
- **AND** Building Locally content lives in `docs/contributing.md`
- **AND** FAQ content lives in `docs/faq.md`, linked from the README

#### Scenario: All four platforms are named

- **WHEN** a reader views `readme.md`
- **THEN** GitHub Actions, GitLab CI, Gitea, and Forgejo are all named
- **AND** a link to the `examples/` directory is present

### Requirement: Generation is code-generation from a single source on a minimal toolchain

`readme.md` SHALL be composed by a dependency-free Node program
(`docs/build.mjs`, Node standard library only) from `config/version.json`, with
no MDX, `mdx-bundler`, or `pnpm` docs workspace. The four `examples/*.yml` SHALL
be produced by `cue export` from CUE definitions, using only CUE's evaluation/
export surface (not `cue cmd`, `tools/file`, or `@embed`). `config/version.json`
SHALL remain the single source of truth, validated by `config/schema.cue`.
`windows.md`, `docs/contributing.md`, `docs/faq.md`, and `LICENSE.md` SHALL be
static committed markdown.

This guardrail is load-bearing: it is what keeps the docs/examples consistent
with the manifest (no stale versions the reader would notice) and valid by
construction (no malformed example a user would copy), while removing the heavy
MDX/`pnpm` dependency tree whose upkeep the maintainer otherwise pays for in
lockfile churn and CVE remediation (the project already carries an `mdx-bundler`
CVE workaround). Reintroducing a template engine or a heavy doc framework would
restore that cost and the staleness/supply-chain risk this guardrail removes.

#### Scenario: The MDX/pnpm toolchain is absent

- **WHEN** the repository is built
- **THEN** there is no `docs/src` pnpm workspace, MDX source, or `mdx-bundler`
  dependency
- **AND** `mise run docs` generates outputs using only `node` and `cue`

#### Scenario: Relocated docs are static markdown

- **WHEN** the repository is built
- **THEN** `docs/contributing.md` and `docs/faq.md` are static committed markdown
  and are not regenerated by `mise run docs`
